## An Introduction to RISC-0 :


RISC Zero is a zkVM, which means it is a virtual machine that lets us prove that a program ran correctly without asking everyone else to rerun it. In simple terms, we write a program, run it inside the zkVM, and obtain a proof that others can verify. In this project, that is useful because we want to prove game events while keeping the secret board hidden.

In RISC Zero, the main program we care about is the **guest program**: the code whose execution we want to prove. To prove that a program executed correctly in this zkVM, we need the following parts:
- A **guest** program compiled into an ELF binary.
	- The guest can **read**, **write** private output to the host, and **commit** public output to the journal.
- The "**executor**" runs the ELF binary and records the session.
	- A *host* provides the environment that runs this executor for the guest program.
- The "**prover**" checks and proves the validity of the session, outputting a *receipt*.
	- A *receipt* contains a **journal** for public outputs and a **seal**, which is what lets us verify that the computation was correct.
	- The *host* above is also the one that executes the guest program and generates the receipt through the executor.

Anyone with a copy of this receipt can verify the correct execution of the guest program.

>[!Rule]
The prover is not the guest; the prover runs the guest, which is the program being proved.

You may remember that every zero-knowledge system needs "constraints". If you studied SNARKs, that is the first question you would ask: **where are the constraints?**
Here, the constraints are generated by default while executing the guest code, because it runs on a virtual machine that simulates a CPU and uses instructions that are all constrained.
![Pasted image 20260302132250.png](/assets/img/Pasted%20image%2020260302132250.png)

The output of the VM's executor is a *session*. This session is the complete record of the computation that happened, and it is divided into segments.

Finally, each proof has what we call an "*Image ID*," which is the Merkle root of the ELF binary loaded into memory. It identifies *which* program was executed and provides a verifiable identity.

# 1. Game Rules and Disclosure Policy : 

- We have **two** participants that should have two (10x10) grid, one where he will place his ships and one where he will use it to hit "the opponent".
	- The placement of ships should be validated, and should not be allowed to change. (*This must be part of the game rules, not of the proving system)
- Everyone has **5 ships** 
	- The carrier (5 squares)
	- Battleship (4 squares)
	- Cruiser (3 squares)
	- Submarine (3 squares)
	- Destroyer (2 squares)
- The initial placement should be **vertical OR horizontal** (not diagonal), and all the ship should be included **inside of the grid**, and **none should be overlapping**.
- The first who should start call a shot on the grid grid of shots, and the opponent say if it is a hit or a miss
	- If it is a hit, a red flag is raised in the place.
	- If it is a miss a yellow one is raised in the place.
- If a ship is hit on all its presence on the board, it sink. The type of the ship sunk **must be announced**.
- He who sink the entire fleet will win.

This means the disclosure policy is embedded directly into the rules of play. What remains secret is **the exact relationship between ship locations and ship types until a ship is actually sunk**; more broadly, the unconfirmed structure of the defender's board must stay hidden throughout the game. What must be proved is the correctness of each hit-or-miss declaration and, when relevant, the claim that a ship of a given type has sunk. Some structural inference from the public history of hits is acceptable, because players can naturally learn patterns from previous rounds, but that should never disclose the exact position of ships that are still afloat. Replay attacks and leakage through error messages are concerns for the proof-system mechanics, not the game logic itself. Observers are treated as potential attackers: they may see the attacks, the announcements of sunk ships, and the proofs attached to hit-or-miss outcomes, but they should never see either player's full board.

Before moving forward, We must denote the threats and how would they come to be. This is called threat modelling and it is a crucial step into the making of any protocol.
It is difficult to do two separate (important) things in the same time, therefore I chose to make this implementation on ==two phases== :
1. The **basic proof system**, here we ensure that the proofs are generated and verified correctly. ==players themselves cannot cheat because the game engine prohibit them of doing so, and this is mainly due to the fact that they are running in one process==. the main vulnerability vector is the *memory* of the game, as it can destroy the secrecy if exploited. but we will not concern ourselves with this as **it goes beyond cryptography, although for a real system it must be addressed**.
2. The **complete system**, and this will come soon. This system should enable "free will" participants, (they can deny or accept the hits) that communicate through a secure channel, enable PvP gameplay. of course we would have an orchestrator that enables the game rounds, but **it will not oblige the players to do something, for instance : a player can position his ship in a incorrect way**, and this will not be accepted after a verification.

Let's start now exploring how each proof should work :
#### First step -> Populating the Grid 

#### Proof 1 : a valid Grid :

Each player should prove that his grid is actually correct, which means that we claim that :

- **There is No overlapping ship** : No ship has a common square with another ship.
- **AND there is No ship off-grid** : No ship has a square that is outside the limits of the grid (10x10).
- **AND No diagonal ship** : The squares should either move by (i+n , j) XOR (i , j+n).

And not only that, but each player should **commit** to the grid, so we can ensure that it is not changed during the game.

A valid question to ask here, as in any ZK project, is: **who is the prover, and who is the verifier?** In a **zkVM-based system**, the prover is **the party that runs the guest program inside the zkVM and generates the proof of execution**; the VM itself is the proving environment, not the prover as an actor. The verifier is **whoever checks the resulting receipt** against the expected **Image ID** and public journal. In our setting, the proof is about the **correctness of a hidden computation**, not merely the declaration of a player, and **the game engine** is the first entity that verifies it, even though the attacker and any observer could verify it as well.
So what should we give to the zkVM as a witness ? For this proof, it will have the grids of both players so it can verify their validity (needless to say that we trust it for that).

Now, coming back to the proofs : 
- The diagonality of a ship is impossible *by design*, because -as explained before- the player does not place ships cell by cell, but follows a model that makes it impossible to place diagonal ships.
- Checking for overlap is done by calculating the number of occupied cells, which should be 17 = 5 + 4 + 3 + 3 + 2, and by passing over each one of them while "drawing a mark" that should remain unique.
- Checking for an off-grid ship is done by a simple check of each ship's cells and by comparing them with the limits (10 for the column and 10 for the row).

Okay, we actually got a proof. But it is not over yet. We should "commit" to this board. 
For that, we would use the most basic (and RISC-0 zkVM-friendly) commitment : a SHA-256 hash that would be verified each round in order to assert that the grid has not been changed. 
But using a raw SHA-256 would be dangerous, as a player may "break it" with enough repetitions. It is for this reason that we add a "blinding factor" in order to obfuscate the grid.
By doing that, we actually achieve the two properties needed by a commitment : **binding (using the hash) and hiding (with the blinding factor)**.

Now that this is clear, we should think about how we can implement the "attack engine" as part of the "Game Engine" so that we can prove it :


#### How can we attack ? 

Basically, each player would have a turn where he can direct fire at a specific cell, guessing that a ship must be there (or eliminating this possibility), without seeing the grid of his opponent. 
To maintain secrecy and help us produce a proof later, we will have "logs" where we can maintain a history of what happened during previous turns. These logs are : 
- **A public attack log** : which would contain every coordinate ever targeted, regardless of result (it has `(coordinate, result)`), with its main purpose **to prevent a player to call an attack on the same cell twice**, this would prevent us from doing a redundant proof later on. This is a public, append-only log, and it is the main input for the coming proofs.
	We also use it to construct a *hit log* that only contains the cells that have been hit, in order to help prove whether a ship sank or not.
- **A public sunken ships log** : where we would add the sunken ships (we would need that for a later proof).

A round would proceed as follows : 
1. Attack Declaration (from the attacking player)
2. Hit/Miss Declaration and Proof (Done by the Game Engine, and NOT the defender. In this implementation, we would not give him a chance to deny a hit; instead, a proof would be generated automatically).
3. If a hit is declared -> Sinking Declaration (Also done by the game engine).
4. Verifying if a game is over.
5. If the game is not yet over : closing the turn, and the game engine changes the position of the players. 


After an attack is declared on a cell (within range, of course), the proving mechanism is fired : 

#### Proof 2 : A valid Hit (or a sorry miss)

The claim being proven here is: *"The attack at `(x, y)` against my committed board -which I committed to in Round 0- resulted in HIT or MISS."*
In order to prove that, the defender will provide : 
- The 5 ship descriptors `(row, col, orientation)`.
- The blinding factor.
These will be received by the zkVM, and the circuit would receive in addition : 
- The stored commitment C for this player.
- The attack coordinate `(ax, ay)`.
- ==The current round number.== (This is a very important information, as a malicious player may attempt "a replay attack").

Our circuit would first check that the commitment recomputed from the witness matches C, in order to establish "a chain-of-trust" that ensures the uniqueness of the grid. Then we proceed in the "intuitive way" (because there is no need to get fancy) : for each ship, the circuit derives all cells from its descriptor and checks whether `(ax, ay)` -the attack coordinate- matches any of them. If yes, it is a HIT; otherwise, it is a MISS.

The public journal issued by the zkVM would contain, at the end of the proof, the commitment C, the attack coordinate `(ax, ay)`, the result (a HIT or a MISS), and the round number (to prevent the replay attack).

Now for the verification :
- The receipt verifies against the hit/miss guest program ID (remember that it is the actual cryptographic fingerprint that proves that a *specific* program has been executed. In this case : a specific attack -the one ordered- has been run, which means that we can "take the word" of the zkVM).
- The commitment in the journal matches the stored commitment for this defender.
- The coordinate in the journal matches the declared attack for this round.
- The round number in the journal matches the current round.


We are not finished yet, because we want to tie all this to the next question : "If a ship is hit, was it enough to sink it ?"
Before we start, bear in mind that : **The game engine would force the defender to claim that a ship sunk, and this claim will be verified**. This is done once a hit is registered and verified.

#### Proof 3 : A fallen fellow (or not)

Let's deconstruct what we want : we want to prove that a ship sunk because of a series of hits on its position, or not. These are two different things, and therefore two different claims. 
The first one is : *"Every cell of ship X has been hit by a distinct shot in the public transcript, against my committed board."* The second one is : *"For each of  the ships still alive on the grid, at least one of its cells has not yet been hit."* 

Isn't it enough to prove that a ship sank ? No, it is not. We must ensure that every possible outcome is taken into consideration, and proven as well.

In order to prove that, we would need the **Hit Log**, which would be derived by filtering the public transcript, and the defender would send to the zkVM : the 5 ship descriptors and blinding factor, and for each cell of that ship, the index into the hit log where that cell appears.

The circuit inside the VM would **recompute** the commitment and check it again (yes, it is redundant, but we will not risk an alteration here). It would check each cell of the ship declared sunk and assert that all its cells are actually present in the hit log.

The output journal would then contain the commitment C, the ship index (which identifies the type unambiguously via the fixed-order array), and the length of the public transcript at the time of proof (also in order to prevent the use of this exact log or proof for a later round or a later game).

The game engine would then verify : 
- The Guest program ID (always checking it to ensure that the program that has been executed IS the program that we wanted)
- That C matches stored commitment.
- That transcript length matches current length, and that the ship index is also the same.

When this is done, the engine would record this ship index in the list of sunk ships for this player, and announce the sinking publicly.

Now let's turn to the 2nd claim : **No ship sunk in this turn**. For this, the defender has to provide the 5 ship descriptors and blinding factor (like always, this is becoming tedious! but it is necessary), and for each ship, the index of one of its cells that has not yet been hit. 
The circuit will therefore check if the commitment match is correct (this is also becoming tedious, isn't it ? that's the price of security). And for each ship that is not present in the sunken ships log, we would check whether it has at least one cell that is "safe." Of course, we have previously proven the correctness of the sunken ships log.

The public journal would contain : 
- The receipt against the ImageID of the program (just like the other proofs).
- The commitment C.
- The current transcript length (no ship has been added, therefore it is correct).
- Already sunk indices match — consistency with prior rounds.


#### Proof 4 : Was it the last bullet ? We Won!

Finally, the chain continues : if a ship is registered as sunk, the game engine would force the defender to capitulate and verify the "defeat condition" and, of course, the win for the attacker. The easiest way to verify that we won is to have **17 registered hits, equal to the number of cells**.

Since the attack log prevents double-hitting the same cell (now you get why we prevented that :) ), and we have ensured that the public transcripts are not tampered with, **17 confirmed hits means every one of the 17 ship cells has been hit exactly once. All ships are necessarily sunk.**

The engine announces the winner. The game ends.
It was quite a ride.

Well, this is true if the winning condition is actually met; otherwise, we would stay in a loop...
